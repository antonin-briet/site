🌐 Caddy + DNS Cloudflare

Reverse proxy HTTPS avec certificats Let's Encrypt automatiques via challenge DNS Cloudflare. Accès unifié à tous les services via sous-domaines, exclusivement depuis le réseau Tailscale.

CaddyLet's EncryptCloudflare APIReverse ProxyHTTPSDocker

📋 Contexte & objectifs

Caddy est le reverse proxy qui unifie l'accès à tous mes services sous un domaine personnalisé (votredomaine.dev). Il gère automatiquement les certificats TLS via Let's Encrypt, sans intervention manuelle — c'est ce qu'on appelle le « TLS automatique ».

La difficulté principale venait du fait que le serveur est derrière un VPN sans IP publique exposée. Le challenge HTTP classique de Let's Encrypt est impossible dans ce cas — j'ai donc utilisé le challenge DNS via l'API Cloudflare, ce qui permet d'obtenir un certificat valide même sans port ouvert.

⚠️ Contraintes

  • IP privée Tailscale — impossible d'utiliser le challenge HTTP classique Let's Encrypt
  • Domaine .dev impose HTTPS obligatoire (HSTS préchargé)
  • Services accessibles uniquement via VPN, jamais depuis internet

🔧 Difficultés rencontrées

  • Let's Encrypt refusait les challenges HTTP car l'IP n'était pas routable publiquement
  • Caddy ne pouvait pas résoudre les DNS ACME à cause du conflit avec AdGuard
  • Exposition involontaire des services avant fermeture des ports dans UFW

✅ Solutions apportées

Image Caddy enrichie du plugin slothcroissant/caddy-cloudflaredns. Token API Cloudflare injecté via variable d'environnement. UFW configuré pour bloquer les ports 80/443 depuis internet, seul le trafic sur l'interface tailscale0 est autorisé.

💡 Ce que j'ai appris

La mise en place du challenge DNS m'a fait comprendre en profondeur le protocole ACME et les alternatives au challenge HTTP. J'ai également mieux saisi la notion de HSTS et pourquoi certains domaines imposent HTTPS de façon permanente. C'est une compétence directement applicable en entreprise pour sécuriser des services internes.
