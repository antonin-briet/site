Homelab & Cloud
Self-hosted

Face à la dépendance croissante aux services cloud propriétaires (Google, Apple, Microsoft), j'ai choisi de reprendre le contrôle de mes données personnelles en déployant mon propre serveur cloud à domicile sur du matériel recyclé.

L'objectif était triple : souveraineté des données (aucune donnée personnelle sur des serveurs tiers), sécurité maximale (zéro port ouvert sur internet, accès uniquement via VPN WireGuard), et haute disponibilité (stratégie de sauvegarde 3-2-1, redémarrage automatique, monitoring continu). Le tout sur un ancien PC de bureau reconverti en serveur, fonctionnant 24h/24 en consommation minimale.

Ce projet m'a permis d'appliquer concrètement les concepts vus en BTS SISR — réseaux, sécurité, administration Linux, virtualisation de services — dans un environnement réel et entièrement géré de façon autonome.

01 / 08
🔐
Vaultwarden
Production Docker HTTPS 2FA

Déploiement d'une instance Vaultwarden auto-hébergée en remplacement d'un gestionnaire de mots de passe propriétaire. Compatible Bitwarden, accessible uniquement via réseau Tailscale, avec partage sécurisé d'une collection en lecture seule pour un second utilisateur.

Contraintes
  • Accès strictement limité au réseau VPN Tailscale
  • HTTPS obligatoire (imposé par le domaine .dev)
  • Gestion de plusieurs utilisateurs avec droits distincts
  • Migration depuis un gestionnaire existant (export/import CSV)
Difficultés rencontrées
  • Certificats TLS initialement gérés via Tailscale, migrés vers Let's Encrypt via Caddy
  • Sécurisation du token admin (passage en hash Argon2)
  • Restriction de création d'organisations aux seuls administrateurs
Solutions apportées
Reverse proxy Caddy avec DNS challenge Cloudflare pour Let's Encrypt. Token admin hashé via Argon2, signups désactivés, 2FA via application TOTP dédiée. Organisation partagée avec collection en lecture seule pour l'utilisateur secondaire.
02 / 08
🛡️
AdGuard Home
Production DNS DoH Docker

Serveur DNS récursif avec blocage de publicités et de trackers, déployé en tant que résolveur global pour tous les appareils connectés au réseau Tailscale. Bloque environ 21% des requêtes DNS.

Contraintes
  • Doit être le DNS de référence pour tous les appareils Tailscale
  • Upstreams DNS chiffrés (DNS over HTTPS) uniquement
  • Port 53 exposé uniquement sur l'interface Tailscale
Difficultés rencontrées
  • Conflit DNS au démarrage (Tailscale tentait d'utiliser AdGuard avant qu'il soit prêt)
  • Problèmes de résolution DNS sur Windows malgré Tailscale actif
  • Configuration du Global Nameserver dans le dashboard Tailscale
Solutions apportées
Désactivation de accept-dns côté serveur Tailscale pour éviter les boucles. Upstreams configurés en DoH sur Quad9 et Cloudflare. Global nameserver Tailscale pointant vers l'IP Tailscale du homelab.
03 / 08
🌐
Caddy + votredomaine.dev
Production Reverse Proxy Let's Encrypt Cloudflare

Reverse proxy HTTPS unifiant l'accès à tous les services sous un domaine personnalisé (votredomaine.dev), avec certificats Let's Encrypt générés automatiquement via challenge DNS Cloudflare. Les services sont accessibles via sous-domaines, exclusivement depuis le réseau Tailscale.

Contraintes
  • IP privée Tailscale — impossible d'utiliser le challenge HTTP classique de Let's Encrypt
  • Domaine .dev impose HTTPS obligatoire (HSTS préchargé)
  • Services accessibles uniquement via VPN, pas depuis internet
Difficultés rencontrées
  • Let's Encrypt refusait les challenges HTTP car l'IP n'était pas routable publiquement
  • Caddy ne pouvait pas résoudre les DNS ACME (conflit avec AdGuard)
  • Exposition involontaire des services avant fermeture des ports dans UFW
Solutions apportées
Image Caddy enrichie du plugin Cloudflare DNS (slothcroissant/caddy-cloudflaredns). Token API Cloudflare injecté via variable d'environnement pour le challenge DNS. UFW configuré pour bloquer les ports 80/443 depuis internet, en n'autorisant que le trafic entrant sur l'interface tailscale0.
04 / 08
🔒
Tailscale VPN
Production WireGuard Zero Trust MagicDNS

VPN mesh basé sur WireGuard reliant tous les appareils personnels dans un réseau privé sécurisé. Constitue la seule porte d'entrée vers l'infrastructure, remplaçant toute exposition directe sur internet.

Contraintes
  • Aucun port ouvert sur la box internet
  • Le serveur ne doit pas être joignable sans Tailscale actif
  • Partage d'accès contrôlé pour un second utilisateur
Difficultés rencontrées
  • Tailscale essayait d'utiliser AdGuard comme DNS avant son démarrage
  • Windows ne respectait pas les paramètres DNS Tailscale
  • Conflit entre MagicDNS et le DNS local de la box
Solutions apportées
Paramètre accept-dns=false côté serveur pour éviter les boucles. Global nameserver dans le dashboard Tailscale pointant vers AdGuard. Partage granulaire du nœud homelab via la fonctionnalité Share de Tailscale.
05 / 08
💾
Backrest + Rclone
Production Restic Backblaze B2 3-2-1

Stratégie de sauvegarde 3-2-1 : sauvegardes locales chiffrées par service via Backrest/Restic sur le HDD interne, synchronisées chaque nuit vers Backblaze B2 via Rclone. 4 dépôts distincts, snapshots OS via Timeshift.

Contraintes
  • Chiffrement obligatoire des données avant envoi sur le cloud
  • Rclone requiert sudo pour les fichiers appartenant à root (Docker)
  • Backblaze B2 doit être configuré en mode natif B2, pas S3
  • 10 Go de stockage gratuit sur Backblaze B2
Difficultés rencontrées
  • Dépôt Restic non initialisé après reconfiguration des volumes Docker
  • Conflit entre le dossier de données Backrest et le dépôt Restic au même emplacement
  • Les snapshots Timeshift dépassaient le quota Backblaze gratuit
Solutions apportées
Séparation stricte des dossiers : données Backrest d'un côté, dépôts Restic de l'autre. Initialisation manuelle des dépôts via docker exec. Exclusion de Timeshift du sync Rclone (--exclude "timeshift/**"), Timeshift conservé uniquement en local.
06 / 08
🔥
Sécurité réseau
Production UFW Fail2ban SSH

Durcissement complet de l'infrastructure : pare-feu UFW restreignant tout trafic entrant à l'interface Tailscale, Fail2ban contre les attaques brute-force, SSH par clés uniquement, mises à jour automatiques via unattended-upgrades.

Contraintes
  • Aucune exposition directe sur internet
  • SSH accessible uniquement via VPN Tailscale
  • Pas de connexion root possible
Difficultés rencontrées
  • Ouverture involontaire des ports 80/443 lors de l'installation de Caddy
  • Règles UFW redondantes après plusieurs modifications successives
  • Services Docker inaccessibles après fermeture trop stricte des ports
Solutions apportées
Règle UFW unique : Anywhere on tailscale0 ALLOW IN Anywhere. Tout le trafic non-Tailscale est bloqué par défaut. Action1 (RMM cloud) utilisé pour la maintenance à distance sans exposition SSH supplémentaire.
07 / 08
🐳
Dockhand
Production Docker Monitoring Sécurité

Interface web d'administration Docker permettant la supervision en temps réel des conteneurs, la gestion des stacks Docker Compose, le scan de vulnérabilités (Trivy/Grype) et les notifications par email.

Contraintes
  • Accès au socket Docker (/var/run/docker.sock) requis
  • Authentification obligatoire (interface complète sans auth par défaut)
  • Healthchecks à configurer manuellement sur chaque service
Difficultés rencontrées
  • Erreur "Could not query registry" — conteneur sans DNS externe
  • Healthcheck Caddy échouant (redirection HTTP→HTTPS + absence de curl/wget)
  • Mauvaise indentation YAML du healthcheck dans l'éditeur Dockhand
Solutions apportées
Ajout de serveurs DNS externes (1.1.1.1 / 8.8.8.8) dans le docker-compose. Healthcheck Caddy via l'API admin locale (port 2019, HTTP). Scan de vulnérabilités activé avec Trivy et Grype, notifications SMTP configurées.
08 / 08
🏗️
Infrastructure globale
Transversal Docker Compose Debian Réseau

Conception et déploiement de l'architecture complète : organisation des volumes Docker, réseau partagé entre services, gestion des certificats, stratégie de nommage des domaines, optimisation énergétique du serveur (CPU powersave, BIOS).

Contraintes
  • PSU propriétaire HP — migration de boîtier impossible
  • i5-9400F sans GPU intégré — carte graphique obligatoire au démarrage
  • Serveur dans un espace confiné — gestion thermique à surveiller
  • Un docker-compose par service pour une meilleure maintenabilité
Difficultés rencontrées
  • Tentative de retrait de la GTX 1050 — serveur impossible à démarrer sans GPU
  • Réseau Docker partagé entre services pour permettre la communication interne via Caddy
  • Températures à surveiller après installation dans l'espace confiné
Solutions apportées
GTX 1050 conservée (5W idle). Réseau Docker caddy partagé entre tous les services. Service systemd cpupower en mode powersave. BIOS configuré : redémarrage automatique après coupure de courant. Températures stabilisées à ~30°C idle malgré l'environnement confiné.
